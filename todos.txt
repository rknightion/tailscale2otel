1. WHAT THIS PROJECT IS
--------------------------------------------------------------------------------
A lightweight Go service that polls the central Tailscale API
(api.tailscale.com/api/v2) for every available kind of observability data and
exports it as OTEL-native metrics + logs over OTLP. Optimized for Grafana Cloud
but works with any OTEL backend. Reference inspiration: ktranslate
(/Users/rob/repos/ktranslate) â€” sflowâ†’OTEL; same pipeline shape on the TS API.

4. AGREED DESIGN DECISIONS (do not relitigate without reason)
--------------------------------------------------------------------------------
USER decisions: Go single static binary; flow logs as BOTH aggregated metrics AND
  full-fidelity OTEL logs; OTLP push (gRPC/HTTP) + stdout for local debug; scope =
  ALL Tailscale data sources; webhook receiver built but OFF by default; flow-log
  default verbosity per_connection (configurable per_record|off); BOTH poll AND
  stream (Splunk-HEC-style receiver) ingestion (S3/GCS object-reader DEFERRED).

ENGINEERING decisions:
  D1 Hybrid client: official tsclient for snapshots + a thin custom HTTP doer
     (tsapi/logs.go) for the two /logging endpoints. One shared retrying client.
  D2 Official OTEL log SDK (not ktranslate's third-party logs lib).
  D3 Flow-log "tail" hazard: never poll up to now; poll [checkpoint, now-lag].
     nextWindow() (collector/window.go) does cold-start lookback, lag, max-window cap.
  D4 Device identity = METRIC ATTRIBUTES (host.name/host.id/os.type/os.version/
     tailscale.user), NOT a per-device Resource. (See P2-4 closed decision.)
  D5 Synchronous instruments (Float64Counter.Add / Float64Gauge.Record).
  D7 Checkpoint store: in-memory default + file-backed opt-in (atomic temp+rename).
  D8 Streaming ~= polling in content; receiver emulates Splunk HEC (POST
     /services/collector/event, `Authorization: Splunk <token>`). TS does NOT
     document the exact HEC envelope â†’ parser is DEFENSIVE; envelope still to be
     pinned by a live capture (S4-10).
  D9 [S4-1] Event name uses the NATIVE log.Record.EventName (log v0.20.0+), not the
     old "event.name" attribute. (S4-9 live-verified: native EventName â†’ Loki key
     `event_name`, identical to the old attribute â†’ dashboards unaffected. DONE.)
  D10 [S4-4] Cumulative temporality pinned explicitly on both OTLP metric exporters
     (Grafana Cloud/Mimir accept cumulative ONLY; deltaâ†’HTTP 400). KEEP histogram
     _bucket/_sum/_count flattening (matches Mimir's classic storage; do NOT
     reconstruct native/exponential histograms â€” would lose explicit buckets).
  D11 [S4-3] webhookâ†”audit cross-dedup is BEST-EFFORT and OFF by default
     (webhook.dedup_audit_events). Shared dedup.Set, first-seen-wins; the
     typeâ†”action map / subjectID / time-bucket need live tuning (S4-10/S4-11). It
     defaults to NO dedup whenever it can't confidently map (never over-suppresses).
  D12 [S4-7] Flow-log volume guard caps LOG records per window with a VISIBLE
     logs_dropped counter (no silent caps); metrics are NEVER capped. Stream body
     is bounded (decompressed) â†’ 413 on overflow (anti-OOM/zip-bomb).
  - semconv naming: dotted lowercase, no `_total` suffix in source, UCUM units
    (By, {packet}, {flow}, {event}, {record}, s, d, 1). Low-cardinality dims on
    metrics; full 5-tuple / high cardinality only on log records.

CLOSED decisions (do NOT reopen without a new reason):
  - P2-4 PER-DEVICE RESOURCE = DEFERRED (user-confirmed). One Resource per
    MeterProvider, so true per-device Resource needs N providers + a target_info
    series per device = churn/cost for ~zero new capability (identity is already on
    every metric via D4). Node-level metrics are delivered by the nodemetrics
    scraper (label-based "instance"), not per-device Resource.
  - NODE-METRICS identity = the "instance" LABEL, NOT a per-node Resource.
  - P2-6 (no native EventName) = RESOLVED by S4-1 (D9).
